Model Context Protocol

用一套标准，
让 AI 接上整个世界

MCP 是一个开放协议，规定了 AI 应用（如 Claude、ChatGPT）如何统一地连接外部的数据、工具和工作流——让模型不再只是"会聊天"，而是能读取真实信息、执行真实操作。

🔌

把 MCP 想成 AI 世界的 USB-C 接口。USB-C 之前，每种设备一根专用线；有了统一接口，一根线插遍所有设备。MCP 对 AI 与外部系统做的，正是这件事。

01 / 它解决的问题

从"每个都要单独接"到"接一次就通用"

假设有 M 个 AI 应用、N 个工具。没有标准协议时，每一对都要写一套专属对接代码——工作量是 M×N。MCP 提供统一"插口"，任何应用和任何工具只要各自实现一次，就能互通——工作量降到 M+N。点下面切换看差别。

02 / 架构：三个角色

Host、Client、Server

MCP 是客户端—服务器架构。关键规则：一个 Server 对应一个 Client，一条专用连接。你的 AI 应用（Host）想连几个 Server，就在内部开几个 Client。点卡片看职责。

Host · 宿主

AI 应用本身

你直接使用的程序。负责协调、管理内部的多个 Client。

例：Claude Desktop · VS Code · Claude Code
Client · 客户端

连接管理器

由 Host 创建，每个只维护与一个 Server 的连接，负责取回上下文。

例：Host 内部的连接实例（用户看不到）
Server · 服务器

能力提供方

对外提供工具、数据、模板的程序。可跑在本地，也可跑在远端。

例：文件系统 · 数据库 · Sentry · GitHub
本地 Server 用 stdio 传输（进程直连）· 远程 Server 用 Streamable HTTP 传输（可服务多个客户端）
03 / 协议的两个层

"说什么" 和 "怎么传"

MCP 把通信拆成两层，各管一件事。你写业务逻辑时基本只关心数据层；传输层多由 SDK 处理。

数据层

定义消息的内容与语义，用 JSON-RPC 2.0 收发请求/响应/通知。包含生命周期管理和核心原语（工具、资源、提示）。这是 MCP 的"灵魂"，也是开发者最常打交道的部分。

传输层

定义消息怎么送达与鉴权。两种方式：stdio（本地进程，零网络开销）和 Streamable HTTP（远程，支持 OAuth 等鉴权）。换传输方式，上层消息格式不变。

04 / 核心：原语 Primitives

双方能相互提供的六种能力

原语是 MCP 里最重要的概念——它规定了双方能给对方什么。Server 提供三种（把能力交给 AI），Client 也提供三种（让 Server 反过来借用 AI 和用户）。点卡片看对应的方法名。

Server 提供 → 供 AI 使用 Client 提供 → 供 Server 回借

举个能串起来的例子：一个数据库 MCP Server 可以同时暴露——用于查询的 Tool、包含表结构的 Resource、以及内置示例的 Prompt。三者配合，AI 就能"看懂库、会查库、按最佳姿势查"。

06 / 一页速记

把这几句记住就够了

// 一句话

  • MCP = AI 的 USB-C：一套开放标准，让任意 AI 应用接入任意外部工具与数据。
  • 价值：把 M×N 的重复对接变成 M+N，一次实现、处处可用。

// 三个角色

  • Host — 你用的 AI 应用
  • Client — Host 内部，一对一连一个 Server
  • Server — 提供工具/数据的程序（本地或远程）

// 两个层

  • 数据层 — JSON-RPC 2.0，管"说什么"
  • 传输层 — stdio / HTTP，管"怎么传"

// 六个原语

  • Server：Tools(做事) · Resources(给数据) · Prompts(给模板)
  • Client：Sampling(借模型) · Elicitation(问用户) · Logging(记日志)
