提示词编写

为 Chat、ChatGPT Work 和 Codex 编写实用的提示词

中文翻译 · 原文：Prompting | ChatGPT Learn

提示词概述

提示词，就是你用来告诉 ChatGPT：你想了解什么、创建什么或修改什么的内容。它可以是一个问题、一条指令，也可以是一个目标。你不需要使用技术化语法，也不必套用僵硬的公式。只需用自己的话开始，查看回复，再通过后续消息逐步调整结果。

简短的提示词往往就足够了。对于规模更大或更重要的任务，请加入真正关键的部分：

  • 目标：ChatGPT 应该做什么？
  • 背景：哪些信息或来源会有帮助？
  • 输出：你需要什么格式、篇幅或详细程度？
  • 边界：哪些内容必须保持不变？ChatGPT 应避免什么，或在采取什么行动前先征求你的意见？

只使用对任务有帮助的部分即可。你不必每项都填，也不需要遵循规定格式。

描述你需要的结果

从结果开始说，而不是先列出一长串详细步骤。如果受众或格式会影响 ChatGPT 应该产出的内容，就把这些信息也写进去。

把这些会议记录整理成一份给项目团队看的简短更新。 把决策和后续步骤放在最前面。

这条提示词说明了要创建什么，以及谁会阅读。如果过程本身很重要，就描述过程；否则，给 ChatGPT 留出空间，让它自行搜索、比较信息并调整方法。

补充有用的背景信息

分享那些可能改变结果的信息。只添加真正相关的来源，并说明 ChatGPT 应从每个来源中提取什么。

  • 如果你希望 ChatGPT 对文档、电子表格、演示文稿或 PDF 文件进行总结、比较、转换，或创建供审核的文件，请附上这些文件。
  • 如果任务依赖视觉信息，请添加截图、示意图或其他图片输入。请明确指出重要区域，不要只让图片自己说明一切。
  • 如果答案依赖最新信息，请要求 ChatGPT 使用网页搜索；如果你需要核查结果，也请要求提供来源。
  • 当相关的聊天或任务需要共享文件、来源或本地文件夹时，请使用项目。

使用已连接的数据源

当 ChatGPT 能访问已连接的数据源时，请说明它应该去哪里查找，以及要找什么。你不需要逐一描述它应该执行的每次搜索。

使用云端硬盘中的最新项目计划，以及项目 Slack 频道里的相关决策和进展，准备一份状态更新。

使用已连接的数据源需要相应的插件；能否使用还可能取决于你的套餐和工作区设置。

使用插件

插件为 ChatGPT 提供可复用的指令，以及连接 Google Drive、Gmail、Slack、GitHub 等工具的能力。只需说明你想要的结果，让 ChatGPT 从可用工具中选择。若要指定插件，请在编辑框中输入 @。

了解插件：在 ChatGPT 中查找、安装和使用插件。

个性化 ChatGPT

对于希望跨聊天和任务持续生效的偏好，请在“设置 > 个性化”中写入自定义指令。仅与当前任务相关的细节，应保留在本次提示词中。

查看个性化设置：设置默认个性、自定义指令和其他应用偏好。

设置能防止实际问题的边界

边界是少量但关键的指令，用来防止 ChatGPT 额外制造工作，或执行你并不打算让它做的操作。如果改错某个细节会让结果无法使用，或者你希望在内容影响其他人之前先审核，就应加上一条边界。

  • 保持已批准的日期和预算数字不变。
  • 只使用提供的来源。缺少信息时请标明，不要猜测。
  • 所有建议都必须控制在规定预算内。
  • 把消息准备成草稿，不要发送。

重点写清最重要的一两个边界即可。你不需要控制 ChatGPT 执行的每一个步骤。

让结果可以直接使用

告诉 ChatGPT 你打算如何使用结果。这能帮助它选择合适的篇幅、详细程度和组织方式。

  • 把它整理成一页摘要，让主管能在会前快速浏览。将决策和后续步骤放在最前面。
  • 把这些笔记整理成一封跟进邮件，列出决策、负责人和截止日期。
  • 制作一张清晰的表格，对比计划支出和实际支出，并突出显示超过 10% 的差异。

对于重要工作，可以要求 ChatGPT 做最终检查，例如确认每项行动都有负责人和截止日期，或标出它无法验证的信息。之后，在使用或分享结果前，请你自己再检查一遍。

用后续消息改进结果

第一条提示词不必完美。查看结果后，再具体说明你想修改什么。

让开头更直接，保留证据，并把建议移到背景说明之前。

你可以补充遗漏的来源、纠正方向、要求另一个方案，或改变详细程度，而无须从头开始。

中途引导与排队

Codex 已经在工作时，你无需等待当前运行结束，也可以继续发送消息：

  • 中途引导（Steer）会把消息加入当前运行。适合改变方向、补充遗漏的细节或提供新信息。
  • 排队（Queue）会把消息保存到下一次运行。适合应在当前工作完成后再处理的后续事项。

在 ChatGPT 桌面应用中，可前往“设置 > 通用 > 后续消息行为”选择默认方式。排队的消息会显示在编辑框上方，你可以编辑、重新排序、发送或删除。该设置还会显示快捷键，让你能针对单条消息临时使用另一种方式，而不改变默认设置。

在 Codex CLI 中，Codex 工作时按 Enter 可引导当前轮次，按 Tab 可将消息排到下一轮。详情参见交互快捷键。

把各个部分组合起来

如果要利用已连接的数据源制作项目更新，一条完整的提示词可以这样写：

为周一的管理层会议准备一页项目状态更新。使用云端硬盘中的最新项目计划，以及项目 Slack 频道里的相关决策和进展。 开头先列出管理层需要做出的决策和后续步骤。总结进度、风险、负责人和截止日期。保持已批准的日期和预算数字不变。标出相互冲突或缺失的信息，不要发送或发布任何内容。 结束前，请检查每个后续步骤是否都有负责人和截止日期。

这条提示词涵盖了目标、背景、输出和边界，并要求进行最终检查，但没有把每个步骤都规定死。

使用语音听写

在 ChatGPT 桌面应用中，当编辑框可见时按住 Ctrl+M，然后开始说话。ChatGPT 会把你的语音转写到编辑框中，你可以在发送提示词前检查并编辑文字。

Chat 的提示词示例

Chat 适合用于提问、构思想法、起草内容和处理日常决策。先说明你想得到的结果，只有当细节会改变答案时再补充。

理解一个主题

向从未投资过的人解释复利是如何运作的。 使用一个具体示例，并解释你引入的所有金融术语。

起草和完善文字

起草一封友好的邮件，说明我因为要出行而谢绝这次邀请。 控制在 120 字以内，并为将来的活动保留机会。

比较选项

为一名每年出国旅行两次的用户比较这两个手机套餐。 用表格展示重要差异，然后推荐其中一个并说明取舍。

制定实用计划

规划五顿工作日晚餐，每顿制作时间不超过 30 分钟。避免花生，在不同餐食中重复利用食材，最后提供一份合并后的购物清单。

为 Work 编写提示词

Chat 适合快速提问、简短改写、头脑风暴和轻量草稿。Work 则适合需要利用不同来源或工具、包含一系列步骤、会执行修改，或需要产出较大型交付物的任务。

对于 Work 任务，请描述你需要的结果，提供源材料，说明受众，并解释你将如何审核工作。要求 ChatGPT 制定计划、收集所需信息、创建文件，并在完成前进行检查。

高效使用 Work

Work 适合耗时或周期性任务，也适合创建可复用的成品文件。即使某项任务消耗较多额度，只要能节省时间、提高质量或帮助你做出重要决策，就仍然可能值得。

先从一项你可以审核的结果开始：

  • 只包含相关来源，并在适当时限制日期范围。
  • 明确受众、输出格式和期望篇幅。
  • 将必做工作与可选的改进或润色分开。
  • 如果方法很重要，就要求先制定计划。在 ChatGPT 发送、发布或修改他人依赖的信息前，要求获得你的批准。
  • 如果任务开始执行你已不再需要的工作，请缩小范围或停止任务。

检查第一版结果，完善指令；当流程行之有效时，重复使用它。

把源材料制作成成品文件

使用附带的季度报告制作一份管理层简报和一套六页演示文稿。 受众是高管团队。开头列出他们需要做出的三项决策；区分报告中的事实和你的分析；每个数字都要标注来源文件；完成前检查简报和幻灯片内容是否一致。

研究一项决策

为一家 50 人的公司研究三款客户支持平台。使用当前来源比较定价、安全性、集成能力和迁移工作量。 交付一份建议备忘录，其中包含链接、假设，以及签订合同前我们需要回答的问题。

协调一次发布

根据附带的产品简报制定发布计划。包括时间表、负责人、依赖项、风险、公告草稿、客户常见问题和发布日检查清单。在制作最终文件前，标出所有尚未决定的事项。

对于周期性工作，先在普通任务中完善提示词。输出稳定可靠后，再从该任务中安排定时工作。如果每次定时运行都应该从一个新任务开始，则创建独立的定时任务。

为 Codex 编写提示词

当你希望 ChatGPT 处理代码、代码库或开发工具时，请使用 Codex。一条实用的 Codex 提示词应说明你想要的行为，指出相关代码或复现步骤，保留重要约束，并说明如何验证改动。

对于多步骤任务，如果你希望 Codex 在编辑前先调查并提出方案，可在应用编辑框中输入 /plan。当“目标模式（Goal mode）”可用时，可在计划之后使用 /goal 设置一个持续目标。当前命令列表请参见应用斜杠命令。

如何阅读这些示例

每个工作流程都包括：

  • 适用场景，以及最合适的 Codex 使用界面（IDE、CLI 或云端）。
  • 带有示例用户提示词的步骤。
  • 背景说明：Codex 会自动看到什么，以及你应该附加什么。
  • 验证：如何检查输出。
注意：IDE 扩展会自动把你打开的文件加入上下文。在 CLI 中，请明确提及路径，或者使用 /mention 和 @ 路径自动补全来附加文件。

Codex 会在一个限制文件和网络访问的沙箱中运行本地命令。如果任务需要跨越该边界，Codex 会按照你的审批策略，在继续前请求批准。

解释代码库

当你正在熟悉一个项目、接手某项服务，或尝试理解协议、数据模型或请求流程时，使用此工作流。

IDE 扩展工作流（本地探索最快）

  1. 打开最相关的文件。
  2. 选择你关心的代码（可选，但推荐）。
  3. 向 Codex 输入：
解释请求是如何流经所选代码的。 请包括： - 涉及的每个模块的职责简述 - 数据在哪里进行验证，以及验证哪些内容 - 修改时需要留意的一两个“易踩坑点”

验证：

  • 要求提供一张可以核查的流程图或检查清单：
    用编号列表总结请求流程中的各个步骤，然后列出涉及的文件。

CLI 工作流（适合保留操作记录和命令）

  1. 启动交互式会话：
    codex
  2. 附加文件（可选）并输入：
    我需要理解这项服务所使用的协议。读取 @foo.ts 和 @schema.ts，解释数据结构及请求/响应流程。重点说明必填字段与可选字段，以及向后兼容规则。

背景说明：你可以在编辑框中使用 @ 插入工作区文件路径，或使用 /mention 附加特定文件。

修复错误

当你能在本地复现异常行为时，使用此工作流。

CLI 工作流（复现与验证的紧凑循环）

  1. 在仓库根目录启动 Codex：
    codex
  2. 向 Codex 提供复现步骤，以及你怀疑的文件：
错误：在设置页面点击“保存”后，有时会显示“已保存”，但更改并未持久化。 复现步骤： 1) 启动应用：npm run dev 2) 前往 /settings 3) 切换“启用提醒” 4) 点击“保存” 5) 刷新页面：开关恢复原状 约束： - 不要更改 API 结构。 - 保持修复最小化；如果可行，添加回归测试。 先在本地复现错误，再提出补丁并运行检查。

背景说明：

  • 由你提供：复现步骤和约束（这些比高层描述更重要）。
  • 由 Codex 提供：命令输出、发现的调用位置，以及触发的所有堆栈跟踪。

验证：

  • 修复后，Codex 应重新执行复现步骤。
  • 如果项目有标准检查流程，请要求它运行：
    修复后，运行 lint 和最小范围的相关测试套件。报告所运行的命令和结果。

IDE 扩展工作流

  1. 打开你认为存在错误的文件，以及离它最近的调用方。
  2. 向 Codex 输入：
    找出导致显示“已保存”但更改未持久化的错误。提出修复方案后，告诉我如何在界面中验证。

编写测试

当你想明确界定测试范围时，使用此工作流。

IDE 扩展工作流（基于所选代码）

  1. 打开包含该函数的文件。
  2. 选中定义该函数的代码行。从命令面板选择“Add to Codex Thread”，把这些代码加入上下文。
  3. 向 Codex 输入：
    为这个函数编写单元测试。遵循其他测试中使用的约定。

背景说明：“Add to Codex Thread”会提供所选代码行（即“行号”范围），以及已打开的文件。

CLI 工作流（在提示词中描述路径和行范围）

  1. 启动 Codex：
    codex
  2. 在提示词中写出函数名：
    为 @transform.ts 中的 invert_list 函数添加测试。覆盖正常路径和边界情况。

根据截图制作原型

当你想把设计稿、截图或界面参考转成可运行原型时，使用此工作流。

CLI 工作流（图片 + 提示词）

  1. 把截图保存到本地（例如 ./specs/ui.png）。
  2. 运行 Codex：
    codex
  3. 把图片文件拖入终端，附加到提示词中。
  4. 继续补充约束和结构：
根据这张图片创建一个新仪表板。 约束： - 使用 React、Vite 和 Tailwind。用 TypeScript 编写代码。 - 尽可能准确地匹配间距、字体排版和布局。 输出： - 一个用于呈现界面的新路由/页面 - 所需的小型组件 - 包含本地运行说明的 README.md

背景说明：

  • 图片提供视觉要求，但你仍需说明实现约束（框架、路由、组件风格）。
  • 用文字说明图片无法体现的行为，例如悬停状态、验证规则或键盘交互。

验证：

  • 要求 Codex 启动开发服务器（如果允许），并准确告诉你去哪里查看：
    启动开发服务器，并告诉我查看原型的本地 URL/路由。

IDE 扩展工作流（图片 + 现有文件）

  1. 在 Codex 任务中附加图片（拖放或粘贴）。
  2. 向 Codex 输入：
    创建一个新的设置页面。以附带的截图为目标界面。 遵循该项目其他文件中的设计和视觉模式。

通过实时更新迭代界面

当你希望 Codex 编辑代码时形成紧凑的“设计 → 调整 → 刷新 → 再调整”循环，可使用此工作流。

CLI 工作流（运行 Vite，再用小范围提示词迭代）

  1. 启动 Codex：
    codex
  2. 在单独的终端窗口中启动开发服务器：
    npm run dev
  3. 要求 Codex 修改：
    为落地页提出 2～3 项样式改进。
  4. 选择一个方向，用简短、具体的提示词继续迭代：
    采用方案 2。 只修改页眉： - 让字体排版更具编辑设计感 - 增加留白 - 确保在移动设备上仍然美观
  5. 继续提出聚焦的小范围要求：
    下一轮：减少视觉干扰。 保留布局，但简化颜色并删除多余边框。

验证：

  • Codex 更新代码时，在浏览器中查看变化。
  • 提交你喜欢的更改，撤销不喜欢的更改。
  • 如果你撤销或修改了某项编辑，请告诉 Codex，以免它在处理下一条提示词时覆盖你的改动。

把重构任务委派给云端

当你想利用本地上下文设计方案，再把耗时的实现工作委派给可并行运行的云端任务时，使用此工作流。

本地规划（IDE）

  1. 确保当前工作已提交，或至少已暂存，以便清楚地比较改动。
  2. 要求 Codex 生成重构计划。如果你有 $plan 技能，请明确调用：
$plan 我们需要重构身份验证子系统，以实现： - 拆分职责（令牌解析、会话加载与权限） - 减少循环导入 - 提高可测试性 约束： - 不改变用户可见行为 - 保持公共 API 稳定 - 包含分步骤的迁移计划
  1. 审核计划并协商修改：
    修改计划： - 明确每个里程碑中究竟移动哪些文件 - 包含回滚策略

背景说明：当 Codex 能在本地扫描当前代码（入口点、模块边界、依赖关系图线索）时，规划效果最好。

云端委派（IDE → 云端）

  1. 如果尚未设置，请先设置 Codex 云端环境。
  2. 点击提示词编辑框下方的云朵图标，选择你的云端环境。
  3. 输入下一条提示词后，Codex 会在云端创建一个新任务，并继承现有任务上下文（包括计划和本地源代码改动）：
    实施计划中的里程碑 1。
  4. 审核云端差异，如有需要继续迭代。
  5. 直接从云端创建 PR，或把更改拉到本地进行测试和收尾。
  6. 继续迭代计划中的其他里程碑。

委派到云端的任务会在隔离环境中运行。除非你为该环境启用互联网访问，否则代理阶段默认无法访问互联网。

执行本地代码审查

当你希望在提交或创建 PR 前多一双眼睛检查代码时，使用此工作流。

CLI 工作流（审查工作树）

  1. 启动 Codex：
    codex
  2. 运行审查命令：
    /review
  3. 可选：提供自定义关注点：
    /review 重点关注边界情况和安全问题

验证：

  • 根据审查反馈应用修复，然后重新运行 /review，确认问题已经解决。

审查 GitHub 拉取请求

当你希望获得审查反馈，但不想把分支拉到本地时，使用此工作流。

使用前，请先在仓库中启用 Codex Code Review。详情参见“代码审查”。

GitHub 工作流（评论驱动）

  1. 在 GitHub 上打开拉取请求。
  2. 发表评论，标记 Codex 并明确关注点：
    @codex review
  3. 可选：提供更明确的指令：
    @codex review for security vulnerabilities and security concerns

更新文档

当你需要准确、清晰地修改文档时，使用此工作流。

IDE 或 CLI 工作流（本地编辑 + 本地验证）

  1. 确定需要修改的文档文件，并打开它们（IDE），或使用 @ 提及它们（IDE 或 CLI）。
  2. 在提示词中说明范围和验证要求：
    更新“高级功能”文档，添加身份验证故障排除指南。验证所有链接均有效。
  3. Codex 起草改动后，审核文档并根据需要继续迭代。

验证：

  • 阅读渲染后的页面。
