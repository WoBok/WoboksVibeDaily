桌面端写得好好的渲染逻辑，搬到移动端经常出问题。原因是两个架构差异——UMA 共享内存，和 tile-based rendering。

UMA 架构

桌面端 GPU 有独立显存（VRAM），移动端 GPU 和 CPU 共用系统内存。这意味着：

  • 带宽是 CPU 和 GPU 共享的，渲染时不能占用太多
  • 没有真正的"显存"，所有 texture、buffer、framebuffer 都放在主内存
  • 如果 GPU 渲染时 CPU 也在读写同一块内存，会有竞争

Tile-Based Rendering

桌面端是 immediate mode：每个 draw call 直接渲染到 framebuffer。移动端是 tile-based：先把屏幕分成小块（tile，比如 16x16 像素），每块在片上 tile memory 里渲染完，最后把结果 dump 到主内存。

这个设计的代价是：如果你打断 tile 的连续渲染（插入 load/store 操作），就要付出 tile flush 的代价。

什么时候会 flush？

  • RenderTarget 切换：从一个 RT 切到另一个，前面的 tile 内容要先存回主内存
  • 读 framebuffer 作为纹理：要先把 tile 数据 load 回主内存
  • MSAA resolve：多采样缓冲需要合并成单采样
  • Compute shader 读写 framebuffer：必须先 flush

常见的坑

1. 多余的 RenderTarget 切换

有些自定义 pass 会强制起一个新 RT，渲染完再切回来。每次切换都是一次 flush。

2. 把 scene color 作为输入

后处理里如果需要读 scene color，要先 force flush，再 load。带宽直接翻倍。

3. 不必要的 alpha blending

移动端 alpha blend 需要从 framebuffer 读像素再写回。如果 blend 出现在错误位置，会强制 flush。

UE 移动端的优化路径

UE 的 mobile renderer 设计时已经尽量避免这些坑——比如它会合并 pass、利用 tile memory 做 translucency sorting、延迟部分效果到 base pass 之后。

当你写自定义渲染逻辑时，先想一下：这个操作会不会打断 tile 的连续性？

如果会，能不写就不写。能用 UE 自带的材质/pass 配置解决，就别自己起新 pass。
