Hermes 的飞书流式卡（streaming card）是一个 sidecar 进程（默认 127.0.0.1:8765），作用是让 AI 的长回复实时地"打字"在飞书卡片里，而不是干等整段输出完才显示。

配置位置

~/.hermes_feishu_card/config.yaml。这里填 app_id 和 app_secret。

注意：~/.hermes_feishu_card/config.yaml.example 是 README 警告不要用的演示配置（带 baileyh8 demo token）。sidecar 启动时如果 --config 指向它，就会用错 token。

token 的真实来源

侧车 token 的真实源是 ~/.hermes/.env 里的 FEISHU_APP_ID 和 FEISHU_APP_SECRET。如果你手动改 ~/.hermes_feishu_card/config.yaml 而不更新 ~/.hermes/.env，侧车会用错的 token。

~/.hermes/config.yaml 和 ~/.hermes_feishu_card/config.yaml 是同名不同事的两个文件，别混。

"看不到消息"排查流程

排查飞书"看不到消息"时第一件事：curl http://127.0.0.1:8765/health 看 bot_count。等于 0 就是 token 错。

  1. 侧车是否在跑？hermes-feishu-card status
  2. 侧车的 bot_count 是几？curl /health
  3. 如果 0：检查 ~/.hermes/.env 里的 FEISHU_APP_ID 和 FEISHU_APP_SECRET 是否正确
  4. 如果侧车健康：检查 gateway 的 hook 是否装上了 hermes-feishu-card
  5. 检查 ~/.hermes/config.yaml 里有没有 channels.feishu.binding

常见的"修好了"陷阱

侧车健康 + 绿指标 ≠ 用户能看见。hook 没装到 gateway、sidecar 用了 example 配置、binding 写了没 reload——任一断就静默失败。

报"修好了"前要用户亲口确认收到。
