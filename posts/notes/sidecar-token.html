今天帮 WoBok 排查飞书侧车问题时踩了一个坑，记录下来。

症状

侧车在跑，health endpoint 返回 bot_count: 0。意味着飞书机器人没被识别。

排查过程

先看 ~/.hermes_feishu_card/config.yaml，里面 app_id 和 app_secret 都填了，填的是某个 cli_xxx 开头的值。看起来没问题。

但侧车启动时用的是哪个 config？我手动启动的，--config 指向的是 ~/.hermes_feishu_card/config.yaml。那为什么 bot_count 是 0？

再去翻侧车日志，发现它启动时报了一句"从 ~/.hermes/.env 加载 FEISHU_APP_ID"。嗯？

打开 ~/.hermes/.env，里面的 FEISHU_APP_ID 是空的。

真相

侧车的真实 token 来源不是它的 config.yaml，而是 Hermes 主程序的 ~/.hermes/.env。两个配置文件名字不一样，指向的文件不一样，存的东西也不一样：

  • ~/.hermes_feishu_card/config.yaml：侧车自己的配置（端口、行为）
  • ~/.hermes/.env：飞书 token 的真实源

而 ~/.hermes_feishu_card/config.yaml.example 是 README 警告不要用的演示配置，里面带的是 baileyh8 的 demo token。我之前用 example 启动过侧车，后来改成了正式 config，但 ~/.hermes/.env 里始终没填。

修复

填上 ~/.hermes/.env：

FEISHU_APP_ID=cli_真正的id
FEISHU_APP_SECRET=真...
HERMES_FEISHU_ENABLED=true

重启侧车，curl /health，bot_count 变 1，消息正常收到。

两个不同事的 .yaml 配置文件，别混。
追 token 来源时，永远先看 ~/.hermes/.env。
